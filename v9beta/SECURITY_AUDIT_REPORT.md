### C2 — HT TS 的 DNS 重绑定绕过 / 通过 Fetch-URL 端点实现 SSRF [严重]

| 发现者 | 白盒 #1·白盒 #2 |
|--------|---------------------------|
| **文件** | `app.py`，第 1459-1462 行 |

```python
else:
    # HTTPS：保持原有 URL（无法用 IP 替换，否则 TLS 证书校验会失败）
    actual_url = url
    extra_headers = {}
```

对于 HTTPS URL，已解析的 IP 地址被**丢弃**，原始 URL 被直接传递给 `urllib.request.Request`。攻击者可以利用 DNS 重绑定：

1. 注册一个 TTL 非常短的域名
2. 首先将 DNS 解析为公共 IP — 通过安全检查
3. 将 DNS 切换到 `127.0.0.1` 或云元数据 IP (`169.254.169.254`)
4. `urllib.request` 重新解析 DNS 并连接到内部 IP
5. 响应内容将显示给攻击者

**修复：** 对于 HTTPS，解析 IP 并构建一个自定义 SSL 上下文，该上下文验证原始主机名的证书：
```python
import ssl
ctx = ssl.create_default_context()
ctx.check_hostname = True
ctx.server_hostname = hostname
# 使用 IP 地址和此 SSL 上下文构建请求
```

### C3 — NaN 余额充值业务逻辑绕过 [严重]

| 发现者 | 黑盒 #1 |
|--------|-------------|
| **文件** | `app.py`，第 1085-1098 行 |

```python
amount = float(amount_str)       # NaN 被接受
if amount <= 0:                  # NaN <= 0 → False
    return error
if amount > 100000:               # NaN > 100000 → False
    return error
# NaN 通过了所有验证！
```

**利用：** 发送 `amount=NaN` 会绕过正数验证，因为 IEEE 754 NaN 的比较总是返回 `False`。这将用户余额（SQL `balance + NaN`）损坏为 NULL。

```bash
curl -b "session=<cookie>" -X POST http://127.0.0.1:5050/recharge \
  -d "amount=NaN&csrf_token=<token>"
```

**修复：** 在 float 转换后添加 `math.isnan(amount)` 检查。使用基于整数的余额（存储分而不是元）。

### C4 — 密钥暴露 (会话伪造) [严重]

| 发现者 | 黑盒 #1 |
|--------|-------------|
| **文件** | 环境变量 / `/proc/<pid>/environ` |

Flask 密钥作为环境变量传入 (`FLASK_SECRET_KEY="test-security-audit-key-32chars-min"`)，并通过 `/proc/<pid>/environ` 暴露。拥有此密钥后，攻击者可以伪造任何用户的会话 cookie：

```python
from itsdangerous import URLSafeTimedSerializer
s = URLSafeTimedSerializer("test-security-audit-key-32chars-min")
cookie = s.dumps({"csrf_token": "x", "uid": 1, "session_version": 1})
```

**修复：** 保护环境变量。使用密钥管理服务。立即轮换密钥。

## 高风险问题 (9)

### H1 — 登录速率限制的竞态条件（暴力破解绕过）

| 发现者 | 白盒 #1 |
|--------|-------------|
| **文件** | `app.py`，第 592-612 行 |

`record_rate_failure()` 中的读取-修改-写入序列不是原子的。并发请求可以在任何线程写回之前读取相同的计数，从而允许攻击者通过在 20 个并发请求中发送错误密码来超过 `MAX_ATTEMPTS` (5)。

**修复：** 在速率限制操作周围使用 `threading.Lock()`，或使用 Redis 进行分布式速率限制。

### H2 — 0.0.0.0/8 范围在 SSRF IP 阻止列表中未被完全阻止

| 发现者 | 白盒 #1 |
|--------|-------------|
| **文件** | `app.py`，第 1293-1294 行 |

仅阻止了 `0.0.0.0`，未阻止整个 `0.0.0.0/8` 范围。在许多内核中，`0.1.2.3` 路由到本地主机并绕过了 `is_loopback` 检查。

```python
if ip_str == "0.0.0.0":   # 仅这一个地址被阻止
```

**修复：**
```python
if ip in ipaddress.ip_network("0.0.0.0/8"):
    return True
```

## 中风险问题 (13)

### M1 — 图像验证绕过 Polyglot GIF/PHP

| **文件** | `app.py`，第 400-428 行 |

`validate_image_content()` 仅检查文件头。Pillow 打开图像但不重新编码。PHP 代码可以附加在有效图像数据之后，并通过所有检查。

**修复：** 使用 Pillow 重新编码图像（`img.save(output, format=...)`）而不是按原样保存原始字节。

---

### M2 — `/page` 端点缺少身份验证检查

| **文件** | `app.py`，第 1234 行 |

所有受保护的路由都检查 `session.get("uid")`，但 `/page` 不检查。目前仅提供静态 `help.html`，但如果添加敏感页面，将成为一个旁路。

---

### M3 — 管理员角色存储但从未强制执行

| **文件** | `app.py`，第 100 行 |

数据库中有一个 `role` 列，但**没有** `@admin_required` 装饰器。所有经过身份验证的用户（admin 和 user）对所有功能具有完全相同的访问权限。

---

### M4 — `index.html` 中存在休眠的存储型 XSS (`| safe`)

| **文件** | `templates/index.html`，第 44 行 |

`{{ page_content | safe }}` 禁用了自动转义。目前，没有路由设置 `page_content`，因此不可利用。任何未来使用此变量的代码都会立即创建一个存储型 XSS。

**修复：** 移除 `| safe` 过滤器。

---

### M5 — `allocate_uid` 和注册中的 TOCTOU 竞态条件

| **文件** | `app.py`，第 146-178 行和第 835-845 行 |

UID 分配（SELECT → 检查 → INSERT）和注册用户名检查（检查 → 插入）在单独的事务中运行。并发注册可能会选择相同的 UID 或通过重复的用户名检查。

---

### M6 — 速率限制是进程内且非共享的

| **文件** | `app.py`，第 467 行 |

`LOGIN_ATTEMPTS` 是一个内存中的字典。使用多个 Gunicorn workers 时，每个进程都有自己的计数器。

---

### M7 — 注册端点没有速率限制

| **文件** | `app.py`，第 775-854 行 |

与登录不同，`/register` 没有速率限制，允许攻击者创建数千个账户并耗尽 UID 池。

---

### M8 — 余额类型混淆（浮动点数与整数列）

| **文件** | `app.py`，第 1085、1104 行 vs 第 101 行 |

输入被解析为 `float`，然后存储到 `INTEGER` 列中。SQLite 会截断值，导致潜在的会计差异。

---

### M9 — 注册接受用户名中的空字节和危险字符

| **发现者** | 黑盒 #2 |
|--------|-------------|

用户名如 `admin\x00`、`' OR '1'='1` 和 `'; DROP TABLE users; --` 被成功存储。虽然参数化查询阻止了 SQL 注入，但这些用户名可能会在其他上下文中造成问题。

---

### M10 — 登录时用户名修剪不一致

| **发现者** | 黑盒 #2 |

登录会修剪带有空格的用户名，但注册会存储原始用户名。这使得注册带有前导/尾随空格的用户名可能与另一个用户匹配。

---

### M11 — `_is_suspicious_hostname` 中的混合表示法 IP 地址绕过

| **文件** | `app.py`，第 1340-1354 行 |

仅捕获三种模式（纯十进制、纯十六进制、纯八进制）。混合表示法如 `0x7f.0.0.1`（十六进制第一个八位组，其余十进制）在 glibc 系统上解析为 `127.0.0.1`。

---

### M12 — CSP 允许内联样式 (`'unsafe-inline'`)

| **文件** | `templates/base.html`，第 6 行 |

`style-src 'self' 'unsafe-inline'` 允许注入 CSS，这可能被利用进行数据窃取。

---

### M13 — 缺少 `Referrer-Policy` 和 `Permissions-Policy` 头部

| **文件** | `app.py`，第 437-448 行 |

存在安全头部（HSTS、X-Frame-Options、X-Content-Type-Options）但缺少现代头部。

## 结论

该应用程序展示了强大的基础安全工程：参数化 SQL 查询、CSRF 保护、安全密码哈希、会话版本控制和多层文件上传验证。其中 12 个关键安全控制措施已确认有效运行。
